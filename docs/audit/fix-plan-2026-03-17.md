## Phase 1 — Pre-Deployment Blockers

**Timeline:** Immediate (before any production deploy)
**Effort:** ~1 hour
**Validation:** `pnpm validate:all` + manual browser check on /de pages

### 1A. Skip-link i18n (5 min)

**Files:** `apps/web/src/app/[locale]/(landing)/layout.tsx`, `(marketing)/layout.tsx`
**Issue:** "Skip to main content" hardcoded in English. Translation key `accessibility.skipToMain` exists in all 4 locales.
**Fix:** Both layouts are Server Components with `params.locale` available. Import locale-specific strings directly:

```tsx
// In both layouts, replace:
<a href="#main-content" className="skip-link">Skip to main content</a>

// With a lookup using the locale param (no hook needed in Server Components):
const skipLabels = { en: 'Skip to main content', de: 'Zum Hauptinhalt springen', es: 'Saltar al contenido principal', 'pt-BR': 'Pular para o conteúdo principal' };
<a href="#main-content" className="skip-link">{skipLabels[locale] ?? skipLabels.en}</a>
```

**Why not useTranslation:** These layouts are Server Components — React hooks can't be used. A simple lookup from the already-available `locale` param is the correct pattern.

### 1B. CSP PostHog domain (1 min)

**File:** `apps/web/middleware.ts:94`
**Issue:** `connect-src` missing PostHog. Analytics calls silently blocked in production.
**Fix:** Add to the `connect-src` directive:

```
https://app.posthog.com https://*.posthog.com
```

### 1C. PT-BR hero subheadline qualifier (5 min)

**File:** `packages/i18n/translations/pt-BR/landing-b2c.json:4`
**Issue:** "Envie dinheiro para qualquer lugar do mundo" — unqualified, implies universal transfer.
**Fix:** Add "na diBoaS" qualifier: "Envie dinheiro na diBoaS, na hora, de graça. E faça ele crescer do seu jeito."

### 1D. B2B DE footer duplicate crypto disclaimers (15 min)

**File:** `apps/web/src/components/Layout/Footer/MinimalFooter.tsx` — `getDisclosureKeysForLocale()`
**Issue:** Both generic `crypto` key and MiCA `mica` key render for DE/ES, producing two near-identical warnings.
**Fix:** Make the `crypto` key conditional — only add it for locales that don't receive MiCA:

```tsx
// Replace unconditional:
if (keys.crypto) result.push(keys.crypto);

// With:
if (keys.crypto && !['de', 'es'].includes(locale)) result.push(keys.crypto);
```

**Validation:** Check /de/business footer shows one crypto warning, not two.

## Phase 2 — Race Conditions & Timer Cleanup

**Timeline:** Sprint 1
**Effort:** ~1.5 hours (reduced — 3 false positives removed per CTO board review)
**Validation:** Verify no console errors on navigation, no memory leaks in DevTools Performance tab

### 2A. PreDemo timer cleanup (30 min)

**Files:** `PreDemo/screens/LoginScreen.tsx`, `PreDemo/screens/ConfirmationScreen.tsx`
**Issue:** `setTimeout` in `useCallback` with `timerRef` but no `useEffect` cleanup on unmount.
**Fix:** Add a cleanup `useEffect` that clears all stored timer refs on unmount:

```tsx
useEffect(() => {
  return () => {
    timerRef.current.forEach(clearTimeout);
    timerRef.current = [];
  };
}, []);
```

### ~~2B. Service destroy() methods~~ — REMOVED (false positives)

**CTO board correction:** AlertingService (`destroy()` at line 338), ErrorReportingService (`destroy()` at line 468), and MonitoringService (`destroy()` at line 292) all have proper cleanup methods. These were fixed in the original codebase audit (CLAUDE.md Audit Tracker items #3, #4, #5 — all "Done").

**Remaining P2 concern:** These `destroy()` methods exist but no lifecycle mechanism guarantees they are called on page unload. This is architectural debt — if these services are used as singletons initialized once, consider adding `beforeunload` handlers or Provider-level cleanup to ensure `destroy()` runs. Defer to Phase 7.

### 2C. performance-monitor.ts setInterval (15 min)

**File:** `lib/monitoring/performance-monitor.ts:199`
**Issue:** `setInterval` ID not stored — cleanup impossible.
**Fix:** Store the interval ID and add a `destroy()` method:

```tsx
private flushIntervalId?: ReturnType<typeof setInterval>;

private setupPeriodicFlush(): void {
  this.flushIntervalId = setInterval(() => { ... }, this.config.flushInterval);
}

public destroy(): void {
  if (this.flushIntervalId) clearInterval(this.flushIntervalId);
}
```

### 2C-alt. Service lifecycle guarantee (P2 — defer to Phase 7)

**Files:** AlertingService, ErrorReportingService, MonitoringService, breadcrumbManager, analytics/service, web-vitals
**Issue:** All have `destroy()` methods but no lifecycle mechanism guarantees they're called on page unload.
**Deferred fix:** Add `beforeunload` handler or Provider-level cleanup. Low risk for a pre-launch marketing site (services are initialized once per session).

## Phase 5 — Hardcoded CSS Values (Design Token Migration)

**Timeline:** Sprint 2-3
**Effort:** ~4-5 days
**Validation:** `grep` scans return 0 violations + visual regression check on all pages

### 5A. PreDemo.module.css (164 violations) — 1.5 days

The single worst offender. 42 hardcoded font sizes + 122 hardcoded spacing values.

**Approach:**
1. Map each hardcoded value to the nearest design token
2. Create any missing tokens in `design-tokens.css` if needed (e.g., small spacing like 6px)
3. Replace systematically: `font-size: 14px` → `font-size: var(--font-size-sm)`
4. Replace spacing: `padding: 24px` → `padding: var(--spacing-lg)`
5. Visual regression check on /en/demo at both desktop and mobile

### 5B. PreDream.module.css (65 violations) — 1 day

20 hardcoded hex colors (path selector cards) + 10 font sizes + 35 spacing values.

**Hex color fix:** The path selector colors (safety blue, balance purple, growth red) should use semantic tokens:
```css
/* Add to design-tokens.css: */
--color-path-safety: var(--color-blue-100);
--color-path-balance: var(--color-purple-100);
--color-path-growth: var(--color-red-100);
```

Files: FeeTable (16), ScenarioCards (11), ProseSection (8), LanguageSwitcher (4), LegalDocument (3), ProductCarouselDefault (1), and others.

**Priority order:** FeeTable (most visible on landing pages) → ScenarioCards → ProseSection → rest.

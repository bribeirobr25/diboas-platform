## 1. What Does This Project Do?

**diBoaS** (Digital Bank of Autonomous Services) is a unified financial services platform that combines traditional banking, cryptocurrency, and DeFi (Decentralized Finance) strategies into a single product. The platform targets both **consumers (B2C)** and **businesses (B2B)**, offering yield-earning accounts, investment strategies, and treasury management.

### Current Phase: Pre-Launch / Waitlist

The project is currently a **marketing and onboarding site**. No production banking features are live. The live functionality includes:

- **B2C Landing Page** (`/{locale}`) -- Consumer-facing value proposition with 11 content sections, interactive demo, and dream mode simulator
- **B2B Landing Page** (`/{locale}/business`) -- Treasury management pitch for startups, with an interactive yield calculator
- **Waitlist System** -- Full email signup flow with referral codes, position tracking, Kit.com (ConvertKit) integration, and GDPR-compliant deletion
- **Dream Mode** -- Interactive financial projection simulator where users select an investment path (Safety/Balance/Growth) and see projected returns over time
- **PreDemo** -- Interactive banking demo simulating deposit, send, buy, and wallet screens with fee calculations
- **Future You Calculator** -- Compound interest calculator comparing bank savings vs. diBoaS yield strategies
- **40+ Marketing Pages** -- Personal banking, business, learn, rewards, security, help, careers, investors, legal (privacy, terms, cookies)
- **Social Sharing** -- Personalized OG image generation for waitlist position and calculator results
- **Cookie Consent** -- GDPR-compliant banner with HttpOnly cookie storage and consent API
- **Analytics** -- GA4 (consent-gated), PostHog (lazy-loaded behind consent), web-vitals tracking, custom event bus

### Business Model (Planned)

Three subdomains planned for production:
- **diboas.com** -- Marketing site (current implementation)
- **app.diboas.com** -- Consumer banking/investing app (future)
- **business.diboas.com** -- B2B treasury platform (future)

## 3. Where Is the Main Entry Point?

The application has **multiple entry points** due to Next.js App Router conventions:

### Root HTML Entry

```
apps/web/src/app/layout.tsx  (Root Layout - Server Component)
```

This is the **global entry point**. It:
- Sets up `<html>` and `<body>` tags
- Loads the Inter font (self-hosted via next/font/google)
- Injects GA4 script with consent mode (default: deny)
- Adds Organization structured data (JSON-LD)
- Wraps children with `PostHogProvider` and `WebVitalsTracker`
- Reads CSP nonce from middleware via `x-nonce` header

### Root Page

```
apps/web/src/app/page.tsx  (Root Page - Server Component)
```

Performs a **redirect** to `/{DEFAULT_LOCALE}` (i.e., `/en`), ensuring all URLs have a locale prefix.

### Request Entry (Middleware)

```
apps/web/middleware.ts  (Edge Middleware)
```

Runs on every request before layouts:
- Generates per-request CSP nonce (crypto.randomUUID)
- Detects locale from URL path
- Redirects unlocalized paths (e.g., `/about` -> `/en/about`)
- Sets `Content-Language` and CSP headers

### Route Group Layouts

```
apps/web/src/app/[locale]/(landing)/layout.tsx   -- Landing pages (minimal nav)
apps/web/src/app/[locale]/(marketing)/layout.tsx  -- Marketing pages (full nav + footer)
```

Both layouts:
- Load common i18n messages server-side
- Wrap children with `LocaleProvider`, `I18nProvider`, `PageErrorBoundary`, `WaitingListProvider`
- Add skip navigation link, `CookieConsent`, `ScrollDepthTracker`

### Package Entry Points

```
packages/i18n/src/index.ts    -- Main i18n exports
packages/i18n/src/server.ts   -- Server-safe exports (no React)
packages/i18n/src/client.ts   -- Client-safe exports ('use client')
packages/ui/src/index.ts      -- Design system (Button, cn utility)
```

### Build Entry

```
turbo.json            -- Orchestrates build order across workspaces
next.config.js        -- 378-line config with webpack optimization, Sentry, security headers
package.json (root)   -- Monorepo scripts (dev, build, validate, audit)
```

### Web App Integration (`apps/web/src/lib/i18n/`)

#### config-translator.ts (192 lines)
Bridge between configuration objects and translations:

#### pageNamespaceLoader.ts (70 lines)
Server-side utility for page-specific namespaces:
- `loadPageNamespaces(locale, namespaces[])` -- Loads and flattens multiple namespaces
- Smart prefix mapping: `'home'` -> `'marketing.pages.home'`, `'personal/account'` -> `'marketing.pages.personal.account'`
- Landing pages use own prefix: `'landing-b2c'` -> `'landing-b2c'`

### Data Flow

```
Browser Request
  → Middleware (detect locale from path/cookie/Accept-Language)
  → Layout (Server Component)
    → loadMessages(locale, 'common') + flattenMessages()
    → I18nProvider with common messages
  → Page (Server Component)
    → loadPageNamespaces(locale, ['page-specific-namespace'])
    → PageI18nProvider (merges parent + page messages, memoized)
  → Client Components
    → useMessage('common.buttons.getStarted')
    → useTranslation().formatMessage({ id: 'key' })
```

### Locale Switching

```
User selects language
  → Cookie set (locale-{namespace}, SameSite=lax, 30-day cache)
  → Client: Update locale context, remount I18nProvider
  → Next request: Middleware reads cookie, redirects to new locale path
```

### Translation Validation

`packages/i18n/scripts/validate-translations.js`:
- Validates all locales against reference (`en`)
- Checks: missing files, orphaned files, missing keys, orphaned keys
- Allowlist for locale-specific keys (e.g., Brazil legal disclaimers)
- Run: `pnpm validate:translations`

### Page-Level i18n Pattern

Every page follows this pattern:

```tsx
// page.tsx (Server Component)
export default async function Page({ params }: LocalePageProps) {
  const { locale } = await params;
  const messages = await loadPageNamespaces(locale, ['namespace']);

  return (
    <PageI18nProvider pageMessages={messages}>
      <PageContent />       {/* Client component using hooks */}
    </PageI18nProvider>
  );
}
```

`PageI18nProvider`:
- Memoizes merged messages to prevent context instability
- Merges parent (common) + page-specific messages
- Validates parent context
- Development-only error logging


Translated at render time via `useConfigTranslation()`.
